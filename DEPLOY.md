# Deploying Find My Trial to Render

This deploys the whole app as **one always-on Docker web service** that serves the
FastAPI API and the built React frontend on a **single origin** (so the secure,
`SameSite=Strict` session cookie works with no cross-site relaxation).

Files that make this work (already in the repo):
- [`Dockerfile`](Dockerfile) — builds the React app, then runs FastAPI serving it.
- [`render.yaml`](render.yaml) — Render Blueprint (service + env vars).
- Corpus auto-fetch — on first boot the app downloads the 33 MB trial CSV from
  `FMT_TRIALS_CSV_URL` (it is intentionally **not** committed to git).

> **Demo only.** This is hardened for a public *demo with synthetic / de-identified
> charts*. Do **not** enter real patient data: production PHI use additionally
> requires a BAA with your LLM provider, audit logging, and a persistent user store.

---

## 1. Push the repo to GitHub

The app code has no secrets in it (`.env`, the corpus, and PHI fixtures are all
gitignored), so a **public** repo is simplest — it makes the corpus download in
step 2 work without authentication.

```bash
git add -A
git commit -m "Add Render deploy (Docker single-origin) + de-id fix"
git branch -M main
# create an empty repo on github.com first, then:
git remote add origin https://github.com/<you>/find-my-trial.git
git push -u origin main
```

## 2. Upload the trial corpus (once)

The 33 MB `backend/data/trials.csv` can't live in git. Host it as a **GitHub Release
asset** (stored outside the git tree):

1. On your repo → **Releases** → **Draft a new release**.
2. Tag `corpus-v1`, title anything.
3. **Drag `backend/data/trials.csv`** into the "Attach binaries" box → **Publish**.
4. Right-click the uploaded `trials.csv` → **Copy link**. It looks like:
   `https://github.com/<you>/find-my-trial/releases/download/corpus-v1/trials.csv`

Keep that URL for step 4. (Any public direct-download URL works — S3, R2, etc.)

## 3. Create the Render service

1. Go to **dashboard.render.com** → **New** → **Blueprint**.
2. Connect your GitHub repo. Render reads [`render.yaml`](render.yaml) and proposes
   the `find-my-trial` web service. Click **Apply**.

## 4. Set the secret env vars

In the service's **Environment** tab, fill the values marked `sync: false`:

| Key | Value |
|---|---|
| `FMT_ADMIN_PASSWORD` | a strong password — this is your login |
| `FMT_OPENROUTER_API_KEY` | your `sk-or-v1-…` key |
| `FMT_TRIALS_CSV_URL` | the Release asset URL from step 2 |
| `FMT_CORS_ORIGINS` | leave for now; set after step 5 |

`FMT_SECRET_KEY` is auto-generated by Render; `FMT_ENV=production` and the Gemini
model IDs are already set by the blueprint. Save → Render builds and deploys.

## 5. First boot & verify

First deploy takes a few minutes (Docker build + `npm run build`). On boot the app
downloads the corpus (~a few seconds) and builds the index. Then:

- Visit `https://<your-service>.onrender.com/health` → expect
  `{"ok":true,"trial_count":10000,"llm_enabled":true,"degraded_mode":false}`.
- Open the root URL, log in with `admin` / your `FMT_ADMIN_PASSWORD`.
- (Optional) set `FMT_CORS_ORIGINS` to your `https://<your-service>.onrender.com`
  URL and save.

## Operational notes

- **Memory.** The in-memory index needs ~300 MB; Render's **free plan is 512 MB**,
  which fits but has little headroom. If the service is OOM-killed / restarts under
  use, bump to a plan with ≥1 GB RAM in **Settings → Instance Type**.
- **Cold starts (free plan).** The free service spins down after ~15 min idle; the
  next request cold-starts (re-download corpus + rebuild index, ~10–30 s). A paid
  plan stays always-on.
- **Sessions are in-memory.** A restart (deploy, cold start, OOM) logs everyone out.
  Fine for a demo; a persistent session store is the next step for real multi-user.
- **Updating the corpus.** Upload a new Release asset and point `FMT_TRIALS_CSV_URL`
  at it (or delete the file and let it re-fetch).
